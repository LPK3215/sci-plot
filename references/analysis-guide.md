# Project / Paper / Code Analysis Guide

## Analysis Objective

Deeply understand any research-related content (papers, projects, algorithms, code, documents) provided by the user, and extract the "core information that can be illustrated."

---

## Paper Analysis Checklist

### Step 1: Quick Overview
- Paper title, authors, publication date/venue
- One-sentence core contribution
- What problem does it solve

### Step 2: Method Breakdown
- Overall input→process→output flow of the method
- Break down each core module/component
- Internal mechanism of each module
- Key differences from existing methods

### Step 3: Experiment Analysis
- Most important experimental results (not all results)
- Comparison with baselines
- The number/chart that best demonstrates the method's effectiveness

### Step 4: Identify "Illustratable"
- Method overall architecture → Overview diagram (1 page)
- Each core innovation mechanism → Mechanism breakdown diagram (1 page each)
- Key experimental results → Data/comparison chart (1-2 pages)
- Unimportant or text-suffices content → Skip

---

## Project Code Analysis Checklist

### Step 1: Project Structure
- Project purpose/goal
- Core directory structure
- Main entry files

### Step 2: Core Modules
- Responsibilities of key modules/classes
- Dependency relationships between modules
- Data flow direction

### Step 3: Algorithm Logic
- Core algorithm input/output
- Main steps and decision points
- Key data structures

### Step 4: Identify "Illustratable"
- System architecture → Architecture diagram (1 page)
- Core algorithm flow → Flowchart (1-2 pages)
- Data flow/pipeline → Data flow diagram (1 page)
- Module relationships → Component relationship diagram (1 page)

---

## Algorithm Analysis Checklist

### Step 1: Algorithm Elements
- Input format and constraints
- Output format and meaning
- Core idea (one sentence)

### Step 2: Algorithm Steps
- Step-by-step breakdown of algorithm flow
- Key operations at each step
- Time complexity / space complexity

### Step 3: Baseline Comparison
- What method is it better than, and why
- Key innovation points

### Step 4: Identify "Illustratable"
- Algorithm overview flow → Flowchart (1 page)
- Core innovation steps → Detail diagram (1-2 pages)
- Baseline comparison → Comparison chart (1 page)
- Complexity/performance → Data chart (1 page)

---

## Document Analysis Checklist

### Step 1: Document Positioning
- Document type (technical doc / API doc / design doc / whitepaper)
- Core topic

### Step 2: Structured Information
- Architecture/design overview
- API/interface list
- Data model
- Key processes

### Step 3: Identify "Illustratable"
- System architecture → Architecture diagram
- API call flow → Flowchart
- Data model relationships → ER diagram / relationship diagram
- Business processes → Swimlane diagram / flowchart

---

## Analysis Self-Checklist

After analysis, verify each item:

- [ ] Can you explain what this thing does in one sentence?
- [ ] What are the core innovations/highlights (1-3)?
- [ ] Is the overall flow/architecture line clear (input→process→output)?
- [ ] Is each key module's internals understood?
- [ ] What is the most compelling result/data?
- [ ] Which content truly needs diagrams, and which can be text-only?
